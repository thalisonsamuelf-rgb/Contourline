## Sumario Executivo

A Contourline opera hoje com silos de dados completos: Totvs (ERP), RD Station (CRM), contourline-db (banco auxiliar), ferramentas de IA isoladas no Marketing, e zero integracoes entre eles. Este documento define a arquitetura que vai conectar tudo isso em uma plataforma centralizada de dados e inteligencia.

A arquitetura proposta segue o modelo **Hub-and-Spoke simplificado**: um hub central de integracao (N8N) conecta todos os sistemas fonte a um data warehouse operacional (Supabase/PostgreSQL), com visualizacao via Metabase. Todas as decisoes foram tomadas com um unico criterio predominante: **o time de TI da Contourline tem 3 pessoas e precisa operar tudo isso sozinho**.

**Stack escolhida:**
- **Hub de Integracao:** N8N (self-hosted)
- **Data Warehouse / BI Central:** Supabase (PostgreSQL) — ja em uso pela empresa
- **Visualizacao:** Metabase (self-hosted, conectado ao Supabase)
- **Hospedagem:** Railway ou Render (PaaS gerenciado)
- **Custo mensal estimado:** R$350-900/mes

## 2. Estrategia de Integracao

### 2.1 Padrao de Integracao por Par de Sistemas

| Sistema Origem | Sistema Destino | Padrao | Frequencia | Justificativa |
|---------------|----------------|--------|-----------|---------------|
| **Totvs** | Supabase (BI) | Batch ETL | Diario (06:00) | API Totvs tem rate limits; dados financeiros nao precisam de real-time; batch e mais confiavel para time pequeno |
| **RD Station** | Supabase (BI) | API Polling | 4x/dia (06:00, 10:00, 14:00, 18:00) | Leads precisam de freshness razoavel; API REST v2 suporta polling eficiente |
| **contourline-db** | Supabase (BI) | Batch ETL | Diario (05:30) | SQLite local, volume baixo, sync diario suficiente |
| **Google Ads** | Supabase (BI) | API Polling | Diario (07:00) | Metricas de ads consolidam D-1; API Google Ads e batch-oriented |
| **Meta Ads** | Supabase (BI) | API Polling | Diario (07:00) | Mesmo raciocinio do Google Ads |
| **Supabase (BI)** | Metabase | Conexao direta | Real-time | Metabase conecta direto no PostgreSQL; zero integracao necessaria |
| **WhatsApp Business** | Supabase (BI) | Webhook (futuro) | Event-driven | Quando implementado, mensagens entram via webhook no N8N |
| **Eleven Labs** | Supabase (BI) | API Polling (futuro) | Semanal | Baixa prioridade; metricas de uso apenas |

### 2.2 Por Que Batch e Nao Real-Time

Para uma empresa com time de TI de 3 pessoas, a escolha de **batch ETL como padrao principal** e deliberada:

1. **Simplicidade operacional** — workflows batch sao mais faceis de monitorar, debugar e recuperar de falhas
2. **Tolerancia a falhas** — se um batch falha, voce re-executa; em real-time, eventos perdidos sao mais dificeis de recuperar
3. **Custo menor** — sem necessidade de infraestrutura de streaming (Kafka, RabbitMQ)
4. **Adequado ao negocio** — dados financeiros consolidam D-1; leads 4x/dia e suficiente para B2B com ciclo de 60-120 dias
5. **Previsibilidade** — voce sabe exatamente quando os dados sao atualizados

**Excecao futura:** Se o CEO quiser dashboards com dados intraday, podemos adicionar polling mais frequente (1x/hora) para metricas especificas sem mudar a arquitetura.

## 3. Stack do BI Central

### 3.1 Decisao: Supabase (PostgreSQL) + Metabase + N8N

| Componente | Tecnologia | Papel | Justificativa |
|-----------|-----------|-------|---------------|
| **Data Warehouse** | Supabase (PostgreSQL) | Armazenamento centralizado, modelo canonico, API REST automatica, RLS | Ja em uso pela empresa; PostgreSQL e o DB mais versatil; Supabase adiciona auth, RLS e API gratis |
| **Visualizacao/BI** | Metabase (OSS) | Dashboards, relatorios, self-service analytics | Open-source, facil de usar para nao-tecnicos, conecta direto no PostgreSQL, zero custo de licenca |
| **Hub de Integracao** | N8N (OSS) | Orquestracao de workflows ETL, conectores, agendamento | Open-source, visual (low-code), 400+ conectores incluindo Totvs-like e RD Station, self-hosted |
| **Hospedagem** | Railway ou Render | PaaS para N8N e Metabase | Gerenciado (zero DevOps), deploy com 1 clique, scaling automatico, custo previsivel |

### 3.2 Por Que NAO Outras Opcoes

| Opcao Descartada | Motivo da Rejeicao |
|-----------------|-------------------|
| **Snowflake / BigQuery / Redshift** | Enterprise demais. Custo alto, complexidade operacional, time de 3 pessoas nao consegue operar. Overkill para o volume da Contourline (~50K registros/mes). |
| **dbt + Airflow** | Stack moderna de dados, mas exige engenheiro de dados dedicado. Complexidade operacional alta para 3 pessoas. |
| **Power BI** | Licenca paga por usuario, lock-in Microsoft, nao integra facilmente com Supabase. Metabase faz o mesmo gratis. |
| **Looker / Tableau** | Enterprise pricing. R$50-200/usuario/mes. Para 5-10 usuarios, custo mensal de R$500-2000 so em BI. |
| **Apache Kafka / RabbitMQ** | Streaming desnecessario. Batch ETL atende 100% dos casos de uso da Contourline. Complexidade operacional inaceitavel para 3 pessoas. |
| **Fivetran / Airbyte** | ETL gerenciados sao bons, mas N8N ja faz ETL + automacao + webhooks + cron. Uma ferramenta a menos para manter. |
| **Zapier / Make** | Pricing por execucao fica caro rapido. N8N self-hosted e ilimitado. |

### 3.3 Supabase como Data Warehouse — Detalhamento

**Por que Supabase e nao um PostgreSQL puro?**

A empresa ja usa Supabase (confirmado na auditoria 2.1). Usar o Supabase como data warehouse aproveita:

1. **Row Level Security (RLS)** — controle de acesso por perfil direto no banco. CFO ve tudo, gerente ve seu escopo. Zero codigo na aplicacao.
2. **API REST automatica** — cada tabela do Supabase ganha API REST automaticamente. O AIOX e outros sistemas podem consumir dados sem camada adicional.
3. **Auth integrado** — autenticacao de usuarios vem de graca. SSO com Google pode ser ativado.
4. **Realtime subscriptions** — se no futuro quiserem dashboards com atualizacao automatica, Supabase suporta nativamente.
5. **Dashboard admin** — interface web para explorar dados, util para Rafael Henrique e Thalison Samuel (Head de Growth Marketing).
6. **Backups automaticos** — Supabase Pro inclui backups diarios.

### 4.3 Patterns de Uso

**Pattern 1: ETL Batch Padrao**
```
Trigger (Cron) --> Extract (API/DB) --> Transform (N8N Function) --> Load (Supabase INSERT/UPSERT) --> Log resultado --> Alerta se erro
```

**Pattern 2: Data Quality Check**
```
Trigger (Cron pos-ETL) --> Query de validacao --> Se anomalia detectada --> Alerta para Rafael Henrique + Log
```

**Pattern 3: Alerta de Negocio**
```
Trigger (Cron) --> Query no Supabase --> Se condicao (ex: AP vencendo em 3 dias) --> Notificacao para responsavel
```

### ADR-004: Padrao de Integracao — Batch ETL como Default

| Campo | Valor |
|-------|-------|
| **ID** | ADR-004 |
| **Status** | Proposto |
| **Data** | 2026-03-27 |
| **Decisao** | Usar Batch ETL (cron) como padrao de integracao para todos os sistemas, com polling mais frequente apenas para CRM |
| **Contexto** | A empresa opera em B2B com ciclo de venda de 60-120 dias. Dados financeiros consolidam D-1. A API do Totvs tem rate limits. O time de TI nao tem capacidade para operar infraestrutura de streaming. |
| **Alternativas** | (1) Event-driven com Kafka — baixissima latencia mas complexidade operacional inaceitavel. (2) CDC (Change Data Capture) — elegante mas exige infraestrutura adicional (Debezium + Kafka). (3) Webhooks para tudo — depende de cada sistema oferecer webhooks (Totvs provavelmente nao). |
| **Justificativa** | Batch ETL e o padrao mais simples, mais robusto e mais adequado para o volume e latencia da Contourline. Se um batch falha, re-executa. Se precisa de dados mais frescos, aumenta a frequencia do cron. Zero infraestrutura adicional. |
| **Consequencias** | (+) Simplicidade. (+) Robustez. (+) Baixo custo. (-) Latencia minima de minutos (nao real-time). (-) Se o volume crescer muito (improvavel), pode precisar migrar para streaming. |

## Apendice A — Glossario Tecnico

| Termo | Definicao |
|-------|----------|
| **ETL** | Extract, Transform, Load — processo de extracao, transformacao e carga de dados |
| **Batch** | Processamento em lote, executado em horarios agendados |
| **Hub-and-Spoke** | Arquitetura onde um hub central conecta todos os sistemas (spokes) |
| **Data Warehouse** | Banco de dados otimizado para analytics e relatorios |
| **Modelo Canonico** | Modelo de dados unificado que normaliza entidades de multiplas fontes |
| **RLS** | Row Level Security — controle de acesso no nivel de linha do banco de dados |
| **N8N** | Plataforma open-source de automacao de workflows (pronuncia-se "n-eight-n") |
| **Metabase** | Plataforma open-source de BI e visualizacao de dados |
| **Supabase** | Plataforma open-source de backend baseada em PostgreSQL |
| **Railway** | Plataforma de hospedagem PaaS (Platform as a Service) |
| **PaaS** | Platform as a Service — hospedagem gerenciada onde voce nao cuida do servidor |
| **Polling** | Consulta periodica a uma API para verificar se ha dados novos |
| **Webhook** | Notificacao automatica enviada por um sistema quando um evento ocorre |
| **CDC** | Change Data Capture — deteccao automatica de mudancas no banco de dados |
| **DPIA** | Data Protection Impact Assessment — avaliacao de impacto a protecao de dados (LGPD) |
| **DPO** | Data Protection Officer — encarregado de protecao de dados |
| **TCO** | Total Cost of Ownership — custo total de propriedade |
| **ADR** | Architecture Decision Record — registro de decisao de arquitetura |
| **SSO** | Single Sign-On — autenticacao unica para multiplos sistemas |
| **MFA** | Multi-Factor Authentication — autenticacao de multiplos fatores |
