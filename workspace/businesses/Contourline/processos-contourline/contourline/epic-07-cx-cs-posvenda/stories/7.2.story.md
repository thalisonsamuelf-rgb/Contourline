## Descrição

Após o mapeamento da jornada (story 7.1), esta story implementa a automação do onboarding pós-venda no RD Station CRM, usando N8N como hub de integração. O objetivo é que cada novo cliente da Contourline receba uma sequência estruturada de comunicações e ativações nos primeiros 90 dias após a compra — sem depender de memória do time de CX.

O cliente Contourline (médico/profissional de estética, 35-55 anos) comprou um equipamento de alto ticket e precisa: (1) receber boas-vindas formais, (2) ter instalação agendada, (3) completar o treinamento inicial, (4) ter seu primeiro resultado validado. A automação garante que nenhuma dessas etapas seja esquecida.

Stack: RD Station (CRM existente) + N8N (hub de integração) + Supabase (log e rastreamento) + WhatsApp Business (comunicação com cliente).

---

## Acceptance Criteria

- [ ] Fluxo de onboarding automatizado no N8N com no mínimo 6 touchpoints nos primeiros 90 dias
- [ ] Disparo automático de boas-vindas (e-mail + WhatsApp) em até 2h após fechamento do contrato no CRM
- [ ] Agendamento automático de treinamento inicial enviado ao cliente com link de calendário
- [ ] Checklist de onboarding visível no CRM para o time de CX acompanhar progresso por cliente
- [ ] Alerta interno para time CX quando cliente não respondeu a comunicação em 48h
- [ ] Log de todas as comunicações enviadas registrado no Supabase com status (enviado/aberto/respondido)
- [ ] Fluxo testado com ao menos 3 clientes-piloto antes do go-live
- [ ] Documentação do fluxo publicada (diagrama N8N exportado)

## Tasks Técnicas

**T7.2.1 — Levantar campos de dados necessários no RD Station**
- Identificar eventos que disparam onboarding: deal ganho, contrato assinado, equipamento entregue
- Mapear campos disponíveis: nome, e-mail, WhatsApp, equipamento comprado, responsável CX
- Verificar API RD Station para webhooks de saída

**T7.2.2 — Criar fluxo N8N: trigger de novo cliente**
- Webhook receptor de evento "deal ganho" do RD Station
- Parser dos dados do cliente
- Validação de campos obrigatórios

**T7.2.3 — Implementar sequência de touchpoints (90 dias)**
- Dia 0: E-mail boas-vindas + WhatsApp boas-vindas
- Dia 1: E-mail com link para agendamento de treinamento inicial (Calendly ou similar)
- Dia 7: Verificação de agendamento concluído (se não, alerta para CX)
- Dia 14: E-mail "como está indo?" + link para canal de suporte
- Dia 30: Pesquisa rápida de satisfação (1 pergunta NPS)
- Dia 60: WhatsApp proativo de Mariana Normando (Analista de Experiencia e Relacionamento)
- Dia 90: E-mail de fechamento de onboarding + convite para comunidade/próximo passo

**T7.2.4 — Criar checklist de onboarding no CRM**
- Configurar campos customizados no RD Station para rastrear etapas do onboarding
- Cada etapa com status: pendente / enviado / concluído
- Visível no card do deal/contato

**T7.2.5 — Criar alerta interno para CX (cliente não respondeu)**
- Regra: se cliente não abrir e-mail em 48h ou não responder WhatsApp em 72h → notificação interna
- Canal: e-mail para responsável CX ou notificação no CRM

**T7.2.6 — Criar schema e log no Supabase**
- Tabela `cx_onboarding_log` com: cliente_id, etapa, canal, status, timestamp, metadata
- Procedure de INSERT via N8N a cada comunicação enviada

**T7.2.7 — Configurar templates de comunicação**
- E-mails: HTML responsivo, assinatura Contourline, tom profissional mas acolhedor
- WhatsApp: templates aprovados (se usar API oficial)
- Todos os templates com personalização por nome do cliente e equipamento comprado

**T7.2.8 — Teste com clientes-piloto**
- Selecionar 3 clientes recentes com CX acompanhando
- Executar fluxo completo, registrar problemas
- Ajustes e validação final

## Riscos

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| API RD Station com limitações de rate ou webhooks | Média | Alto | Testar API antes de implementar; usar polling como fallback |
| WhatsApp Business API com processo de aprovação lento | Alta | Médio | Começar com e-mail; WhatsApp como segunda fase |
| Templates de e-mail marcados como spam | Baixa | Alto | Usar domínio verificado; seguir boas práticas SPF/DKIM |
| Dados de clientes incompletos no CRM | Alta | Médio | Validação de campos obrigatórios + alerta para CX preencher |
