# Story 7.8 — QA Gate — Validação da Jornada Completa de CX

> **Epic:** 07 — CX / CS & Pós-venda
> **Status:** Draft
> **Agente:** @qa
> **Estimativa:** S (Small — 2-3 dias)
> **Prioridade:** Alta (gate de encerramento do epic)
> **Data:** 2026-03-27

---

## Descrição

Esta story é o QA Gate de encerramento do Epic 07. Após a implementação de todas as stories (7.1 a 7.7), esta story valida que a jornada completa de CX/CS da Contourline funciona de ponta a ponta: desde o momento em que um novo cliente é cadastrado, passando pelo onboarding automatizado, treinamento no LMS, coleta de NPS, alertas de churn, identificação de upsell, até a visualização no dashboard de saúde.

O QA Gate também valida a integridade dos dados, a confiabilidade dos alertas e a experiência do usuário final (Consultoras de CX e clientes). O objetivo é garantir que nenhum "momento de verdade" da jornada do cliente seja interrompido por falha técnica.

---

## Acceptance Criteria

- [ ] Jornada end-to-end testada com ao menos 5 clientes-piloto reais (não simulados)
- [ ] Todos os 7 flows N8N do epic rodando sem erros por 7 dias consecutivos
- [ ] Dados de clientes-piloto presentes em todas as tabelas Supabase relevantes (onboarding_log, lms_progress, nps_responses, churn_scores, upsell_opportunities)
- [ ] Alertas de churn (AMARELO e VERMELHO) testados e chegando para o time correto em < 1h
- [ ] NPS sendo coletado e exibido corretamente no dashboard Metabase
- [ ] Dashboard de saúde da carteira exibindo dados de todos os clientes-piloto
- [ ] Zero dados corrompidos ou inconsistências entre Supabase e CRM (RD Station)
- [ ] Performance: todos os flows N8N executando em < 30 segundos
- [ ] Documentação de operação do sistema publicada para o time TI e CX

## Checklist de Validação

### 7 Quality Checks do QA Gate

**QC1 — Completude de Requisitos**
- [ ] Todos os critérios de aceite do epic validados
- [ ] Cada AC de cada story (7.1 a 7.7) revisado

**QC2 — Integridade de Dados**
- [ ] Nenhum cliente-piloto com dados faltando em tabela obrigatória
- [ ] Timestamps corretos (sem registros com data futura ou muito antiga)
- [ ] Consistência entre IDs de clientes no CRM e no Supabase
- [ ] Zero linhas duplicadas nas tabelas de log

**QC3 — Fluxos de Automação**
- [ ] Flow de onboarding (7.2): disparo em < 2h após deal ganho
- [ ] Flow LMS (7.3): cadastro automático do cliente funciona
- [ ] Flow NPS (7.4): pesquisa enviada no dia correto (D+30/90/180)
- [ ] Flow Churn (7.5): cron diário executando e atualizando scores
- [ ] Flow Upsell (7.6): gatilhos criando deals no CRM
- [ ] Flow Dashboard (7.7): relatório semanal sendo enviado

**QC4 — Alertas e Notificações**
- [ ] Alerta de detrator NPS chega em < 1h para Consultor correto
- [ ] Alerta de churn AMARELO chega para Consultor responsável
- [ ] Alerta de churn VERMELHO chega para Consultor + gerência
- [ ] Alerta de cliente sem resposta (48h) funcionando no onboarding
- [ ] Relatório semanal chegando para todos os destinatários

**QC5 — Experiência do Cliente (ICP)**
- [ ] E-mails de onboarding com personalização correta (nome, equipamento)
- [ ] Link do LMS no e-mail funcionando em mobile (iOS e Android)
- [ ] Pesquisa NPS respondível em 1 clique no e-mail
- [ ] Certificado digital sendo emitido após conclusão do LMS

**QC6 — Experiência do Time CX**
- [ ] Dashboard Metabase carregando em < 5 segundos
- [ ] Filtros de Consultora funcionando corretamente
- [ ] Painel de ações urgentes mostrando apenas alertas reais
- [ ] Time CX consegue usar dashboard sem suporte de TI

**QC7 — Segurança e Privacidade**
- [ ] Dados de clientes no Supabase com RLS configurado (apenas usuários autorizados leem)
- [ ] Logs de comunicação não contêm dados sensíveis em texto plano
- [ ] Credenciais do LMS transmitidas de forma segura (HTTPS)
- [ ] Acesso ao Metabase protegido por autenticação

## Tasks Técnicas

**T7.8.1 — Preparar ambiente de QA com clientes-piloto**
- Selecionar 5 clientes reais (diferentes fases da jornada)
- Garantir dados completos no CRM para cada um
- Documentar estado inicial de cada cliente-piloto

**T7.8.2 — Executar checklist QC1 — Completude**
- Revisar cada AC de cada story
- Documentar qualquer AC não atendido
- Classificar como: PASS / CONCERNS / FAIL

**T7.8.3 — Executar checklist QC2 — Integridade de Dados**
- Queries SQL de validação no Supabase
- Verificar consistência cruzada entre tabelas
- Verificar consistência com RD Station via API

**T7.8.4 — Executar checklist QC3 — Fluxos de Automação**
- Testar cada flow N8N manualmente (trigger forçado)
- Verificar logs de execução do N8N (sem erros, sem timeouts)
- Medir tempo de execução de cada flow

**T7.8.5 — Executar checklist QC4 — Alertas**
- Simular detrator NPS (score 5) e verificar alerta
- Simular cliente com score de churn > 60 e verificar alerta VERMELHO
- Verificar alerta de cliente sem resposta no onboarding

**T7.8.6 — Executar checklist QC5 — Experiência do Cliente**
- Testar e-mails em clientes de e-mail reais (Gmail, Outlook)
- Testar link LMS em iPhone e Android
- Testar pesquisa NPS em mobile
- Solicitar certificado LMS de teste

**T7.8.7 — Executar checklist QC6 — Experiência CX**
- Sessao de uso com Mariana Normando (Analista de Experiencia e Relacionamento) e Nathalia (Analista de CS)
- Medir tempo de carregamento do dashboard
- Registrar problemas de usabilidade

**T7.8.8 — Executar checklist QC7 — Segurança**
- Testar RLS no Supabase (usuário sem permissão não vê dados)
- Verificar HTTPS em todas as URLs
- Verificar que logs não expõem dados sensíveis

**T7.8.9 — Documentar relatório de QA**
- Resultado de cada QC: PASS / CONCERNS / FAIL
- Lista de bugs encontrados com severidade
- Recomendações para sprints futuros

**T7.8.10 — Emitir veredicto do QA Gate**
- PASS: todos os QCs aprovados ou com concerns menores
- CONCERNS: 1-2 QCs com concerns não-críticos documentados
- FAIL: qualquer QC com FAIL em item crítico de dados ou segurança
