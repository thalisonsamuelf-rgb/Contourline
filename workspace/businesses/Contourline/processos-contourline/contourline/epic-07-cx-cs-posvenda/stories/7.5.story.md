## Descrição

A Contourline sofre de "churn silencioso" — clientes que param de se engajar, não renovam equipamentos ou não fazem upsell, sem nunca reclamar formalmente. Sem dados históricos estruturados, um modelo de machine learning puro não é viável agora. A estratégia é um modelo híbrido: regras manuais baseadas em sinais de comportamento no curto prazo, com capacidade de evoluir para IA preditiva conforme os dados se acumulam.

Esta story implementa o motor de alertas de churn usando dados comportamentais disponíveis: engajamento com LMS, respostas a pesquisas NPS, frequência de chamados de suporte, tempo sem interação com CX e sinais do CRM. O modelo de regras é configurável via Supabase, sem necessidade de redeploy.

Stack: Supabase (dados + regras) + N8N (engine de avaliação diária) + Metabase (dashboard de risco) + RD Station (ações de retenção).

## Sinais de Churn Mapeados

| Sinal | Peso | Direção |
|-------|------|---------|
| Dias sem acessar LMS (após cadastro) | Alto | Negativo |
| NPS < 7 | Alto | Negativo |
| Chamados de suporte abertos > 2 no mês | Médio | Negativo |
| Sem resposta a pesquisas (2+ consecutivas) | Médio | Negativo |
| Dias sem comunicação com CX > 60 | Alto | Negativo |
| NPS >= 9 (promotor ativo) | Alto | Positivo |
| Módulos LMS concluídos >= 3 | Médio | Positivo |
| Resposta positiva ao consultor de relacionamento | Médio | Positivo |

## Tasks Técnicas

**T7.5.1 — Modelar sistema de scoring no Supabase**
- Tabela `cx_churn_scores` com: cliente_id, score_atual, score_anterior, faixa_risco, data_calculo, sinais_ativos (JSON)
- Tabela `cx_churn_rules` com: regra_id, descricao, sinal_campo, operador, valor_threshold, peso, ativo
- Tabela `cx_churn_alerts` com: cliente_id, tipo_alerta, data, score_antes, score_depois, acoes_tomadas

**T7.5.2 — Implementar função de scoring no Supabase**
- PostgreSQL function `calculate_churn_score(cliente_id)` que:
  - Consulta todos os sinais relevantes das tabelas do epic (lms_progress, nps_responses, onboarding_log)
  - Aplica pesos das regras na tabela `cx_churn_rules`
  - Retorna score 0-100 e lista de sinais ativos
- Testar com dados sintéticos

**T7.5.3 — Criar workflow N8N de avaliação diária**
- Trigger: cron diário às 7h (horário Brasil)
- Busca todos os clientes ativos no Supabase
- Para cada cliente: chama função de scoring
- Compara score atual com anterior
- Se mudança de faixa de risco: dispara alerta

**T7.5.4 — Definir faixas de risco e playbooks**
- Faixa VERDE (0-30): monitoramento passivo
- Faixa AMARELA (31-60): contato proativo de Mariana Normando (Analista de Experiencia e Relacionamento) em 5 dias uteis
- Faixa VERMELHA (61-100): intervencao urgente de Xavier Ramos (Gerente Regional CS/CX) ou Andreia Almeida (Diretora de Consultoria e Treinamentos) em 48h

**T7.5.5 — Implementar alertas por faixa**
- Alert AMARELO: e-mail para Mariana Normando (Analista de Experiencia e Relacionamento) + notificacao CRM
- Alert VERMELHO: e-mail para Mariana Normando + Xavier Ramos (Gerente Regional CS/CX) + WhatsApp para responsavel
- Cada alerta inclui: nome do cliente, score, lista de sinais que geraram o risco, link para histórico

**T7.5.6 — Criar log de ações de retenção**
- Campo no RD Station para registrar ação tomada após alerta de churn
- Tabela `cx_churn_alerts.acoes_tomadas` atualizada pelo time CX
- Métrica: taxa de clientes resgatados de faixa vermelha

**T7.5.7 — Criar visualização de risco no dashboard (integração com story 7.7)**
- View Supabase `cx_carteira_risco` com distribuição de clientes por faixa
- Dados disponíveis para dashboard de saúde da carteira (story 7.7)

|-------|--------------|---------|-----------|
