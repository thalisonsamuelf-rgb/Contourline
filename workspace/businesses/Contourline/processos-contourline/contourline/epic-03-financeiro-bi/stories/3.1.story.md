# Story 3.1 — Auditoria do Totvs — Mapeamento de Dados Disponíveis via API

## Descrição

O financeiro da Contourline opera com o Totvs como ERP central, mas o CEO quer inteligência financeira com IA sobre os dados do Totvs — sem substituir o sistema. O maior risco do Epic 03 (declarado como probabilidade Alta e impacto Crítico) é: "API do Totvs limitada ou inexistente".

Esta story resolve esse risco antes de qualquer desenvolvimento: auditoria técnica completa do Totvs, mapeando o que está disponível via API, quais dados existem, qual a qualidade dos dados, quais campos são necessários para o dashboard (Story 3.2), relatórios (Story 3.3), cobrança (Story 3.4) e análise de folha (Story 3.6).

O CFO Claudio Kaizer deve ser envolvido desde esta story — ele precisa dar contexto sobre quais dados são críticos e quais processos têm maior potencial de automação.

O plano B para limitações de API (exportação periódica de dados) também é definido aqui.

## Riscos

| Risco | Mitigação |
|-------|-----------|
| API do Totvs não disponível na versão em uso | Plano B via exportação de dados (T3.1.7) — documentar e validar com TI |
| Dados financeiros sensíveis expostos durante auditoria | Auditoria em ambiente de somente leitura, sem extração de dados reais para ambientes externos |
| CFO resistente a compartilhar dados do processo | Posicionar como diagnóstico para melhorar os processos que ele já conhece como ineficientes |
| Totvs customizado pela Contourline — documentação não reflete realidade | Mapear diretamente na instância em uso, não confiar apenas na documentação oficial |

---

## Notas para o Agente

- Esta story é o risco mais crítico do Epic 03 — se a API não funciona, todo o epic muda de abordagem
- Envolver o CFO Claudio desde o início — sua confiança no projeto é fundamental
- O plano B (exportação) é igualmente válido — não é uma falha, é uma estratégia alternativa
- Segurança dos dados financeiros é crítica — nunca extrair dados reais de produção para ambientes de teste não autorizados
