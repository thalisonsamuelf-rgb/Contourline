## Contexto

A operacao logistica da Contourline gera dados financeiros criticos que precisam alimentar o Totvs para analise de margens: custo de frete, custo de importacao, custo de estoque (CMV), custo de avarias e devolucoes. Atualmente, esses dados sao lancados manualmente no Totvs pelo setor financeiro com delay de dias ou semanas, prejudicando a visibilidade do P&L operacional.

Esta story estabelece a integracao bidirecional entre o sistema operacional (Supabase/N8N) e o Totvs, garantindo que dados logisticos e operacionais alimentem automaticamente o modulo financeiro, e que dados financeiros do Totvs (pagamentos a fornecedores, custos de compra) alimentem os dashboards operacionais.

## Objetivo

Automatizar o fluxo de dados entre o sistema operacional (estoque, logistica, recebimento, expedicao) e o Totvs (modulo financeiro), eliminando lancamentos manuais e garantindo visibilidade em tempo quasi-real do custo operacional no P&L.

---

## Acceptance Criteria

- [ ] AC1: Custo de frete por expedicao lancado automaticamente no Totvs dentro de 4 horas apos confirmacao de expedicao
- [ ] AC2: Custo de importacao (frete internacional + impostos) lancado no Totvs apos fechamento de processo de importacao
- [ ] AC3: Calculo de CMV (Custo da Mercadoria Vendida) atualizado automaticamente apos cada expedicao, disponivel no Supabase para dashboard
- [ ] AC4: Divergencias de recebimento (quantidade, avaria) gerando lancamento de ajuste no Totvs automaticamente
- [ ] AC5: Dados de pagamento a fornecedores extraidos do Totvs e disponibilizados no dashboard operacional (prazo medio de pagamento, aging)
- [ ] AC6: Custo de estoque (valor em estoque por SKU) sincronizado diariamente entre Supabase e Totvs
- [ ] AC7: Dashboard financeiro-operacional no Metabase: custo de frete mensal, CMV, custo de importacao e margem operacional estimada
- [ ] AC8: Log de auditoria de todos os lancamentos automaticos no Totvs com status (sucesso/erro) e timestamp
- [ ] AC9: Reprocessamento automatico em caso de falha de integracao: retry com maximo 3 tentativas e alerta ao time financeiro se falhar
- [ ] AC10: Reconciliacao mensal automatica: relatorio comparando valores do Supabase vs Totvs com divergencias destacadas

## Tasks Tecnicas

**T4.6.1** — Mapeamento de campos financeiros: identificar codigos de lancamento, centros de custo e contas do Totvs para cada tipo de custo operacional

**T4.6.2** — Design da integracao: definir quais dados fluem Supabase → Totvs e quais fluem Totvs → Supabase

**T4.6.3** — Implementar tabela Supabase `financial_integrations`: log de todos os lancamentos com status, payload e timestamps

**T4.6.4** — Desenvolver workflow N8N `dispatch-to-totvs-cost`: ao confirmar expedicao, calcular frete e enviar lancamento para Totvs

**T4.6.5** — Desenvolver workflow N8N `import-cost-integration`: ao fechar processo de importacao, compilar custos e lancar no Totvs

**T4.6.6** — Implementar calculo de CMV no Supabase: trigger PostgreSQL que atualiza CMV por SKU a cada movimentacao de saida

**T4.6.7** — Desenvolver workflow N8N `receiving-divergence-to-totvs`: ao registrar divergencia de recebimento, gerar ajuste no Totvs

**T4.6.8** — Desenvolver workflow N8N `totvs-suppliers-sync`: extrair dados de pagamento a fornecedores do Totvs diariamente

**T4.6.9** — Implementar logica de retry: ao falhar integracao com Totvs, retentar 3x com backoff exponencial e alertar via email se falhar

**T4.6.10** — Criar painel Metabase "Financeiro Operacional": custo de frete, CMV, custo de importacao, aging de fornecedores

**T4.6.11** — Implementar job de reconciliacao mensal: workflow N8N no dia 1 de cada mes comparando valores Supabase vs Totvs

**T4.6.12** — Documentacao de integracao: mapa de dados, codigos de lancamento Totvs e procedimento de contingencia manual

## Riscos e Mitigacao

| Risco | Probabilidade | Impacto | Mitigacao |
|-------|--------------|---------|-----------|
| API do Totvs sem suporte a escrita de lancamentos | Media | Critico | Validar capacidade de escrita na API antes de iniciar; alternativa: lancamento via banco de dados direto |
| Lancamentos duplicados em caso de falha parcial | Media | Alto | Implementar idempotencia: cada lancamento tem ID unico verificado antes de reprocessar |
| Codigos de centro de custo Totvs incorretos gerando lancamentos errados | Media | Alto | Sessao de mapeamento com time financeiro antes do desenvolvimento (T4.6.1) |
| Divergencias na reconciliacao mensal gerando work adicional | Alta | Medio | Reconciliacao automatica com relatorio claro reduz esforco manual |
