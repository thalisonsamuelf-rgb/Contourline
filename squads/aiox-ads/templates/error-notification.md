# Error Notification Template

## Metadata

| Field | Value |
|-------|-------|
| **Type** | Error Notification |
| **Generated by** | Any agent (via @ad-midas as concierge) |
| **Reference** | safety-rules.yaml (escalation protocol) |
| **Principle** | Every error has 3 parts: WHAT, WHY, WHAT TO DO |

---

## Standard Format

```
[{SEVERITY}] {short_title}

WHAT: {one_line_description}
WHY: {technical_context}
WHAT TO DO:
  1. {immediate_action}
  2. {follow_up_action}
ESCALATION: {when_to_involve_human}
```

---

## Severity Levels

| Severity | Prefix | Criteria | Agent action |
|----------|--------|----------|--------------|
| CRITICAL | `[CRITICAL]` | Ban risk, restricted account, imminent financial loss | HALT everything, notify human IMMEDIATELY |
| HIGH | `[HIGH]` | Broken functionality, tracking offline, expired token | HALT affected operation, continue others |
| MEDIUM | `[MEDIUM]` | Rate limit, recoverable timeout, performance degradation | Automatic retry, alert if persists |
| LOW | `[LOW]` | Informational warning, non-critical metric out of range | Log, include in next Pulse |

---

## Escalation Rules by Tier

| Severity | Escalation tier | Rule |
|----------|----------------|------|
| CRITICAL | Human Only | Agent STOPS everything and reports. Human decides |
| HIGH | HITL | Agent presents diagnosis and options. Human confirms |
| MEDIUM | Auto (with alert) | Agent tries to resolve. If fails 3x, escalates to HITL |
| LOW | Auto (silent) | Agent logs. Appears in next Pulse/Digest |

Source: `safety-rules.yaml` section `escalation` and `autonomy-tiers.yaml`.

## Common Error Scenarios

### 1. Rate Limit Atingido (MEDIUM)

```
[MEDIUM] Rate limit atingido -- API Meta

O QUE: Requisicoes a API Meta retornando HTTP 429 (Too Many Requests).
POR QUE: Consumo de pontos atingiu o limite do tier atual.
  Tier: Standard (200 pontos/hora). Consumido: 198/200.
  Referencia: rate-limits.yaml
O QUE FAZER:
  1. Aguardar reset do window (proximo reset em {minutos} minutos)
  2. Operacoes de leitura ficam em fila ate window reabrir
  3. Se urgente: priorizar collect-campaign-metrics sobre outras leituras
ESCALATION: Se rate limit persistir apos 2 windows consecutivos,
  verificar se ha outro client consumindo o mesmo token.
```

### 2. Campanha Rejeitada pela Plataforma (HIGH)

```
[HIGH] Campanha rejeitada -- Violacao de politica Meta

O QUE: O anuncio "{ad_name}" foi rejeitado por "{policy_violation_reason}".
POR QUE: {explicacao_tecnica_da_violacao}.
  Codigo: {error_code}
  Politica: Meta Advertising Standards, secao {secao}
  Referencia: https://www.facebook.com/policies/ads/
O QUE FAZER:
  1. NAO resubmeter o anuncio sem correcao
  2. Revisar o texto/criativo com @fiscal (*revisar-rejeicao {ad_id})
  3. Corrigir o elemento especifico que violou a politica
  4. Resubmeter apos aprovacao do @fiscal (HITL: create-ad-api)
ESCALATION: Se 3+ anuncios rejeitados em 24h na mesma conta,
  ativar protocolo de protecao (ver cenario #6: Ban Alert).
```

### 3. Pixel Nao Disparando (HIGH)

```
[HIGH] Pixel sem eventos -- Tracking offline

O QUE: Pixel {pixel_id} nao registrou eventos nas ultimas {horas}h.
POR QUE: Possiveis causas:
  - Pixel removido ou alterado no site (verificar codigo fonte)
  - CAPI (Conversions API) com token expirado ou endpoint alterado
  - Mudanca no site que quebrou o tracking (deploy recente?)
  - Bloqueador de ads no navegador de teste (falso positivo)
  Ultimo evento registrado: {timestamp_ultimo_evento}
  Match rate antes da falha: {match_rate}%
O QUE FAZER:
  1. Verificar se pixel esta no codigo do site (*verificar-pixel)
  2. Testar disparo manual via Meta Pixel Helper (extensao Chrome)
  3. Verificar status do CAPI no Events Manager
  4. Se CAPI: verificar token e endpoint em .env
  5. Se deploy recente: verificar se tag nao foi removida
ESCALATION: Campanhas ativas continuam gastando SEM tracking.
  Se nao resolvido em 2h, pausar campanhas dependentes deste pixel (HITL).
```

### 4. Token Expirado (HIGH)

```
[HIGH] Access token expirado -- API offline

O QUE: META_ADS_ACCESS_TOKEN retornou erro de autenticacao (HTTP 401/190).
POR QUE: O token de acesso expirou ou foi revogado.
  Codigo: OAuthException (code 190, subcode 463)
  Tipo de token: {long-lived/short-lived}
  Expirou em: {data_expiracao}
O QUE FAZER:
  1. Acessar Business Manager > System Users > Gerar novo token
  2. Atualizar META_ADS_ACCESS_TOKEN no .env
  3. Reiniciar MCP server (meta-ads-mcp)
  4. Validar com: teste de leitura basica (*validar-token)
ESCALATION: Todas as operacoes de API estao PARADAS ate novo token.
  Sem token, agentes nao podem coletar metricas nem executar acoes.
  Prioridade: resolver em menos de 1h para nao perder dados do dia.
```

### 5. Erro de API da Plataforma -- 5xx (MEDIUM)

```
[MEDIUM] Erro de servidor Meta -- API instavel

O QUE: API Meta retornando HTTP 5xx em {count} requisicoes consecutivas.
POR QUE: Problema do lado da Meta (servidor/infraestrutura).
  Codigo: HTTP {status_code}
  Mensagem: {error_message}
  Metastatus: {status_metastatus_com}
O QUE FAZER:
  1. Verificar metastatus.com para confirmar incidente na plataforma
  2. HALT todas as operacoes de write (safety-rules.yaml: on_5xx_response)
  3. Operacoes de leitura: retry com backoff exponencial (max 3 tentativas)
  4. Registrar no WAL com status "halted_outage"
  5. Aguardar resolucao da Meta (nao ha acao do nosso lado)
ESCALATION: Se 5xx persistir por mais de 2h, verificar se ha workaround
  via ads-mcp (fallback MCP). Se ambos MCPs falhando, aguardar resolucao.
```

### 6. Alerta de Ban / Restricao de Conta (CRITICAL)

```
[CRITICAL] Alerta de risco de conta -- Multiplas rejeicoes

O QUE: {count} anuncios rejeitados nas ultimas 24h na conta {account_id}.
  Isso eleva significativamente o risco de restricao de conta.
POR QUE: Contas com 3+ rejeicoes em 24h tem probabilidade elevada de
  restricao pela moderacao AI da Meta. Cada rejeicao adiciona um "strike"
  interno que nao e visivel no Ads Manager.
  Rejeicoes:
    - {ad_1}: {motivo_1}
    - {ad_2}: {motivo_2}
    - {ad_3}: {motivo_3}
  Referencia: ban-prevention.md, 05-meta-api-risks-bans.md
O QUE FAZER:
  1. PAUSAR todas as campanhas ativas (ad-midas ja pausou automaticamente)
  2. NAO resubmeter anuncios rejeitados sem correcao completa
  3. Revisar CADA rejeicao individualmente com @fiscal (*revisar-rejeicoes)
  4. Aguardar 24-48h antes de retomar qualquer atividade
  5. Quando retomar: iniciar com 1 campanha, rampagem de 25%
ESCALATION: Situacao ja escalada para Human Only.
  Se conta for restrita: seguir protocolo de appeal (ban-prevention.md secao 3).
  NAO criar nova conta -- efeito cascata pode suspender TODAS as contas do BM.
```

### 7. Conta Banida (CRITICAL)

```
[CRITICAL] Conta restrita -- Acesso limitado ou desativado

O QUE: Conta de anuncios {account_id} foi restrita pela Meta.
  Status: {DISABLED/UNSETTLED/IN_REVIEW}
POR QUE: {motivo_informado_pela_meta}
  Codigo: {error_code_se_disponivel}
  Historico: {resumo_de_rejeicoes_e_warnings_recentes}
  Referencia: ban-prevention.md secao 3 (Se Banido)
O QUE FAZER:
  1. NAO tentar nenhuma operacao nesta conta
  2. NAO criar conta nova no mesmo Business Manager
  3. Documentar evidencias: WAL, action-ledger, screenshots
  4. Preparar appeal com base nas evidencias (Human Only)
  5. Verificar se outras contas no mesmo BM foram afetadas
  6. Se outras contas ok: isolar a conta afetada (remover do BM)
ESCALATION: Human Only. Agentes APENAS documentam e recomendam.
  Appeal e EXCLUSIVAMENTE humano. Prazo tipico de resposta Meta: 24-72h.
  Durante espera: operar em contas alternativas SE existirem e NAO estiverem afetadas.
```

## Error Log Format

Every notified error MUST be logged in the action-ledger:

```yaml
- timestamp: "{ISO8601}"
  type: error
  severity: "{CRITICAL/HIGH/MEDIUM/LOW}"
  error_code: "{codigo}"
  description: "{descricao_curta}"
  resolution: "{resolvido/pendente/escalado}"
  resolved_at: "{ISO8601_ou_null}"
  agent: "{agente_que_detectou}"
```

## Constraints

- Every error follows the 3-part format: WHAT, WHY, WHAT TO DO. No exceptions
- ESCALATION is mandatory at all severity levels
- No emojis in error messages. Errors are serious
- No long paragraphs. Numbered lists for actions
- Direct language. "Token expired" not "an inconsistency was detected in the authentication process"
- CRITICAL and HIGH: immediate notification via @ad-midas (concierge)
- MEDIUM and LOW: logged, appear in next Pulse
- Scenarios not listed here: use the standard format (section 2) and classify severity by impact
